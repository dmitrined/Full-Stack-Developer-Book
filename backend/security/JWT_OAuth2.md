# JWT & OAuth2: Безопасность современных приложений 🛡️

Современные приложения часто используют протоколы на базе токенов для аутентификации и авторизации.

## 1. JWT (JSON Web Token)
JWT — это компактный, безопасный для URL способ передачи данных между двумя сторонами.

### Структура токена:
1. **Header (Заголовок)**: Тип токена и алгоритм подписи (например, HS256).
2. **Payload (Полезная нагрузка)**: Данные пользователя (Claims), такие как `sub` (ID), `role`, `exp` (истечение).
3. **Signature (Подпись)**: Хэш от заголовка и нагрузки, созданный с помощью секретного ключа.

### Схема работы:
- Пользователь отправляет логин/пароль.
- Сервер проверяет их и выдает **Access Token** и **Refresh Token**.
- Клиент сохраняет Access Token (обычно в `localStorage` или `HttpOnly Cookie`) и прикрепляет его к каждому запросу в заголовке `Authorization: Bearer <token>`.

## 2. OAuth2 и OpenID Connect (OIDC)
**OAuth2** — протокол авторизации (дает доступ к ресурсам).
**OIDC** — слой аутентификации над OAuth2 (дает информацию о пользователе).

### Основные роли:
- **Resource Owner**: Пользователь (Вы).
- **Client**: Приложение, запрашивающее доступ (Ваш сайт).
- **Authorization Server**: Сервер, выдающий доступ (Google, GitHub).
- **Resource Server**: Сервер с данными (API).

### Основной Flow (Authorization Code):
1. Клиент перенаправляет пользователя на Google.
2. Пользователь логинится и дает разрешение.
3. Google возвращает **Authorization Code** клиенту.
4. Клиент обменивает код на **Access Token** напрямую у Google (через back-channel).
5. Клиент использует токен для доступа к API.

## 3. Рекомендации по безопасности
- **Всегда используйте HTTPS**: Токены передаются в открытом виде в заголовках.
- **Минимальное время жизни**: Access Token должен жить недолго (например, 15 минут).
- **HttpOnly Cookies**: Лучший способ защитить токен от XSS атак на фронтенде.
- **Валидация на сервере**: Всегда проверяйте подпись и срок годности токена при каждом запросе.
