# Hibernate & ORM: Глубокое погружение 💾

ORM (Object-Relational Mapping) позволяет работать с БД через Java-объекты (Entities). Hibernate — самая популярная реализация спецификации JPA.

## 1. Жизненный цикл сущности (Entity Lifecycle)
Сущность в Hibernate может находиться в одном из 4 состояний:

1. **Transient**: Объект создан через `new`, но еще не связан с сессией Hibernate. В БД его нет.
2. **Managed (Persistent)**: Объект связан с сессией. Hibernate отслеживает изменения и автоматически синхронизирует их с БД (Dirty Checking).
3. **Detached**: Сессия закрыта, объект больше не отслеживается, но в БД он есть.
4. **Removed**: Помечен для удаления. Будет удален из БД при коммите транзакции.

## 2. Проблема N+1 запросов
Одна из самых частых ошибок производительности. Возникает, когда для получения списка из N объектов делается 1 запрос для списка и N дополнительных запросов для связанных данных.

### Решения:
- **Join Fetch**: Самый частый способ. Включает JOIN прямо в JPQL.
  `SELECT u FROM User u JOIN FETCH u.roles`
- **@EntityGraph**: Позволяет декларативно описать, какие связи подгружать жадно.
- **BatchSize**: Группирует запросы по ID.

## 3. Lazy vs Eager Loading
- **LAZY**: Загружает связанные данные только при первом обращении (через прокси). Стандарт для `@OneToMany` и `@ManyToMany`.
- **EAGER**: Загружает данные сразу при получении основного объекта. Стандарт для `@OneToOne` и `@ManyToOne`. 
  *Важно:* Избыточное использование EAGER ведет к просадке производительности.

## 4. Кэширование
- **Первый уровень (L1)**: Кэш внутри `Session` (EntityManager). Включен всегда, работает в рамках одной транзакции.
- **Второй уровень (L2)**: Кэш на уровне всей `SessionFactory`. Работает между сессиями (Ehcache, Redis).
- **Query Cache**: Кэширует именно результаты запроса.
